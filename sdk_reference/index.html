Getting Started

Follow these steps to set up the Battlefield 6 Portal SDK and start creating custom experiences.

  1. 1

    Open the included Godot executable.

  2. 2

    A"Project Selection" window will open. Drag the GodotProject folder into this window to import automatically, or press"Import" to find it manually.

  3. 3

    Open the Battlefield 6 Portal Project and wait for it to start up. This can take a few minutes.

  4. 4

    Click the "Portal Setup" button in the "BFPortal" tab on the right side of the Godot UI and wait for setup to finish. This can take several minutes.

  5. 5

    To begin spatial editing, open a level by opening the "Scene" dropdown and selecting "Open Scene". Open the levels directory and select the level you wish to edit.

Tip: Level geometry will often be above the default camera location in Godot. Look up if you can't find the level.

Key Features

Spatial Editor

Battlefield 6 Portal allows for additive spatial editing of all levels via the included Godot editor. You may add new instances of most objects that appear in a level to that level, and move, rotate, and scale them to build whatever you can imagine. In addition to environmental objects, Portal includes a number of gameplay objects that can be added and modified on every map, such as Headquarters, Spawn Points, Combat Areas, etc. Map edits are exported from Godot and can be attached to Custom Experiences via the Portal Web Builder.

Important Gameplay Objects

Combat Area

Important: CombatArea acts as a vertical ceiling, not a ground-level boundary!

How CombatArea Works:

• Purpose: Prevents players from flying/climbing too high

• Positioning: Placed ~140m above ground (40-50m above highest terrain)

• Height: Typically 100m vertical extent

• Ground Boundaries: Defined by terrain geometry, not CombatArea

Structure:

• CombatArea parent node has no transform (stays at origin)

• CollisionPolygon3D child has the Y-position offset (ceiling height)

• PolygonVolume height extends upward from Y position

Editing Tips:

• Move polygon points: Hold left-click and drag

• Create new points: Ctrl+left-click on edge

• Remove points: Ctrl+right-click on point

• Working Combat Areas are pre-placed on all maps

Example: Terrain at Y=70-104, ceiling at Y=140, height=100 → Volume extends Y=140 to Y=240 (36-70m above ground)

HQ's and Player Spawners

To deploy into a level, you must set up either HQ_PlayerSpawner or PlayerSpawner objects.

HQ_PlayerSpawner:

• Standard Battlefield HQ spawner

• Players manually spawn using their team's HQ

• Assigned to specific team via Inspector Panel

PlayerSpawner:

• Alternate spawning method without HQ

• Script player deployment directly

• Not team-specific, can spawn any players

Both spawner types must be linked to one or more SpawnPoint objects to function. Working HQ examples are pre-placed on all levels.

SpawnPoints

Objects which, when connected to HQ_PlayerSpawners and PlayerSpawners, determine where players are able to spawn into a level.

Area Triggers

  • • OnPlayerEnterAreaTrigger
  • • OnPlayerExitAreaTrigger

Portal Experience Options

Comprehensive settings and mutators available in the Portal Web Builder to customize your experience.

Verified vs Custom Experiences

Verified Experiences

  • ✓ Full XP & Progression - Players earn standard XP and unlock rewards
  • ✓ Limited Modifications - Can modify existing modes but not create entirely new gameplay
  • ✓ No Custom Scripting - Spatial editing only, no TypeScript/Blockly
  • ✓ Portal Publishing - Can be published to portal.battlefield.com
  • ✓ ModBuilder_GameMode: 2 - Experience JSON setting

Custom Experiences

  • ✓ Modified XP - Players earn reduced XP (not full progression)
  • ✓ Full Portal Tools - Spatial editing + TypeScript/Blockly scripting
  • ✓ Custom Gameplay - Create entirely new game modes and mechanics
  • ✓ Custom UI & AI - Build interactive menus and scripted bot behavior
  • ✓ ModBuilder_GameMode: 0 - Experience JSON setting

Choosing Mode Type: Use Verified for traditional Battlefield experiences with full progression. Use Custom for experimental game modes, training scenarios, or creative projects where modified XP is acceptable.

However for more advanced users, and creators looking to build large bespoke experiences, Portal now supports logic scripting via TypeScript.

Advanced Tips & Best Practices

Production-tested techniques and optimizations discovered through extensive development work.

Portal_Reference Layer for Terrain

Critical Optimization: Keep terrain meshes in a separate Portal_Reference layer instead of the Static layer.

Why This Matters:

  • • Terrain data is NOT exported to .spatial.json files (Portal uses base map terrain)
  • • Including terrain in Static layer adds megabytes of unnecessary data to your experience
  • • Portal_Reference layer is automatically excluded from exports
  • • Keeps your experience files lightweight and fast to load

Implementation:

[node name="Portal_Reference" type="Node3D" parent="."]

[node name="MP_Tungsten_Terrain" parent="Portal_Reference" instance=ExtResource("6")]
transform = Transform3D(1, 0, 0, 0, 1, 0, 0, 0, 1, 0, -15.2, 0)

[node name="Static" type="Node3D" parent="."]
# Only custom objects here - NO terrain!

Terrain Height Alignment

Challenge: GLB terrain meshes have internal Y-coordinate baked in, but Portal expects objects positioned relative to Y=0.

Solution - Y-Offset Compensation:

  • • Read terrain bounds from GLB mesh (e.g., Y min = 15.2m)
  • • Apply negative Y-offset to terrain instance (e.g., -15.2m)
  • • This moves mesh DOWN so surface aligns with Y=0 plane
  • • Objects placed at Y=0 now sit correctly on terrain surface

Pro Tip: Use terrain_provider.py to automate height lookups and object snapping to terrain.

Terrain Centering & Rotation

Centering Terrain:

GLB terrain meshes may not be centered at world origin (0, 0, 0). To center terrain for easier object placement:

  • • Read terrain bounds from GLB (AABB bounding box)
  • • Calculate center offset: center = (min + max) / 2
  • • Apply negative offset to terrain transform
  • • Now terrain is centered at world origin (0, 0, 0)

Rotating Terrain:

Some maps require terrain rotation to align with gameplay direction (e.g., north-facing).

  • • Define terrain_rotation in map metadata (degrees)
  • • Convert to radians and apply Y-axis rotation matrix
  • • All placed objects automatically reference rotated coordinate system
  • • Maintains proper north/south/east/west alignment

Snap to Floor/Terrain

Critical Feature: Automatically position objects on terrain surface instead of floating or underground.

How Terrain Snap Works:

  • • TerrainProvider loads heightmap data from GLB mesh
  • • For each object, query terrain height at (X, Z) position
  • • Set object Y coordinate to terrain height
  • • Objects now sit perfectly on terrain surface
Snap to Floor (Y=0)
  • ✅ Fast - no heightmap lookup
  • ✅ Good for flat terrain maps
  • ❌ Objects float on hills
  • ❌ Objects underground in valleys
Snap to Terrain (Dynamic Y)
  • ✅ Perfect terrain alignment
  • ✅ Works on any terrain
  • ✅ No floating/underground objects
  • ⚠️ Requires GLB heightmap

Centering Assets & Objects

Object pivot points may not align with their visual center, causing rotation and placement issues.

Common Centering Scenarios:

1.
Buildings: Often pivoted at corner instead of center. Apply offset for proper rotation.
2.
Trees: Usually centered horizontally, base at Y=0. No adjustment needed.
3.
Vehicles: Check asset catalog for pivot metadata. Some have offset centers for gameplay.
4.
Decals (Lakes): Non-uniform scale often required. Define width/height in mappings.

Asset Mapping & Lookup Strategy

Essential for Map Porting: When converting maps from other Battlefield games (or other games entirely), you need to map original assets to equivalent Portal assets.

The Asset Mapping Problem:

Example: BF1942 map has"Birch_01" tree → Portal has 200+ tree variants → Which Portal tree should we use?

Without mapping, your converted map will have missing objects or placeholders everywhere.

Lookup Strategy Architecture:

1. Create Mappings Database (JSON)

Store mappings in tools/asset_audit/[game]_to_portal_mappings.json 

{
"static_objects": {
"Birch_01": {
"portal_asset":"Birch_01_L",
"category":"nature/trees",
"notes":"Large birch tree"
 },
"Bunker_German_01": {
"portal_asset":"Military_Bunker_02",
"category":"architecture",
"scale": {"x": 1.2,"y": 1.0,"z": 1.2}
 }
 }
}
2. AssetMapper Component

Centralized lookup service for asset translation 

class AssetMapper:
 def map_asset(self, source_type: str) -> str:
"""Map source game asset to Portal asset."""
 mapping = self.mappings.get(source_type)
 if mapping:
 return mapping["portal_asset"]
 # Fallback: return placeholder
 return"Placeholder_Object"
3. Validation & Asset Audit

Check which assets need mappings before conversion 

# Run asset audit to find unmapped assets
python3 tools/validate_map_assets.py --map Kursk

# Output: List of unmapped assets needing attention

Finding Equivalent Portal Assets:

By Name Similarity
  • • Search asset_types.json
  • • Look for similar names (Birch → Birch)
  • • Check visual resemblance in Godot
By Function
  • • What does this object DO?
  • • Bunker → Military cover structure
  • • Find Portal asset with same purpose
By Era/Theme
  • • WW2 → Use era-appropriate assets
  • • Don't mix futuristic with historical
  • • Maintain thematic consistency
By Size/Scale
  • • Match original object dimensions
  • • Store scale adjustments in mappings
  • • Test in Godot for gameplay fit

Pro Tip: Create mappings incrementally. Start with most common assets (trees, rocks, buildings), then fill in gaps. Use find_asset_alternatives.py to discover similar Portal assets automatically.

Asset Management & Optimization

Level Restrictions

  • • Assets in Portal have levelRestrictions
  • • Only use assets approved for your base map
  • • Check asset_types.json before placement
  • • Cross-map assets will fail to load in-game

Asset Variety

  • • Use AssetRandomizer for natural variety
  • • Define tree/rock/shrub variant pools
  • • Prevents repetitive visual patterns
  • • Automatically applied during conversion

Coordinate Systems

  • • Portal uses Godot coordinate system (Y-up)
  • • Right-handed coordinate space
  • • Transform matrices: 3x3 rotation + translation
  • • Use TransformFormatter for conversions

Scale Handling

  • • Portal supports non-uniform scaling
  • • Lake decals use dynamic X/Z scaling
  • • Store scale metadata in mappings JSON
  • • Always test scaled objects in-game

Portal Experience Builder

Critical Experience Structure:

  • 1. Dual Attachment Reference: Spatial attachments MUST appear in BOTH mapRotation[i].spatialAttachment AND root attachments[] array
  • 2. UUID Consistency: Both references must use the SAME UUID or Portal UI won't recognize the attachment

  • 4. Verified Status: Set "ModBuilder_GameMode": 2 for portal.battlefield.com publishing capability
